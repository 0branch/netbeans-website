<div class="sect2">
<h3 id="_what_is_a_node">What is a Node?</h3>
<div class="paragraph">
<p>Nodes are presentation objects.  They have actions, properties and localized display names - they are the place where the architecture meets the human.  Nodes typically wrap some model object and provide human-readable names, icons, etc.  They are at the heart of a lot of NetBeans selection and user interface systems.</p>
</div>
<div class="paragraph">
<p>Nodes are a generic tree structure.  A common use for them is to display <a href="DevFaqDataObject.asciidoc">DataObject </a>s to the user - to represent the user&#8217;s files visually.  Each node has a Children object that can supply a list of child nodes.  Nodes are not visual components, and they do not subclass TreeNode from the JDK - they are more related to the JavaBeans specification, subclassing <code>java.beans.FeatureDescriptor</code>.</p>
</div>
<div class="paragraph">
<p>Nodes are displayed in <em>explorer views</em>.  The Explorer API provides a number of Swing components which take a Node and can display that node and its children - in trees, lists, tree tables, etc.  The property sheet is also an Explorer view - Nodes have properties, which are key-value pairs with localized names.</p>
</div>
<div class="paragraph">
<p>Generally Nodes should <em>represent</em> not <em>be</em> the objects the user is interacting with - if you are putting huge amounts of logic in your Node class, you&#8217;re probably doing something wrong.</p>
</div>
<div class="paragraph">
<p>Nodes form the basis of global selection in NetBeans - each <a href="DevFaqWindowsTopComponent.asciidoc">component in a tab in the ui </a> has an "activated Node".  The system globally tracks what component has focus, and each component typically offers some node as the currently selected node (which can change when the user clicks, etc.).</p>
</div>
<div class="paragraph">
<p>A Node has a <a href="DevFaqLookup.asciidoc">Lookup </a> which you can ask for objects your code is actually interested in.  You never get the selected node and then cast it to some specific Node subclass and do things to that;  the real model objects should be available from the Node&#8217;s Lookup.  This helps to future-proof your code - you can have another module provide the same objects your client code is interested in from <em>its` Node&#8217;s lookup, and the client code never has to change - it&#8217;s just looking for any Node that has what it needs.</em></p>
</div>
<div class="paragraph">
<p>Read about <a href="DevFaqNodeSubclass.asciidoc">how to implement your own Nodes </a></p>
</div>
</div>
