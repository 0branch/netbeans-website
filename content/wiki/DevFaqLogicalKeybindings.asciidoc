<div class="sect2">
<h3 id="_why_should_i_use_d_for_ctrl_and_o_for_alt_i_thought_c_stood_for_ctrl_and_a_stood_for_alt">Why should I use D- for Ctrl and O- for Alt - I thought C stood for Ctrl and A stood for alt?</h3>
<div class="paragraph">
<p>(Wondering what this is about?  See the <a href="DevFaqKeybindings.asciidoc">general FAQ item on keybindings</a>).</p>
</div>
<div class="paragraph">
<p>There should be <strong>no</strong> Alt-bound keyboard shortcuts on Macintosh, <em>ever</em> - it is used on international keyboards as the compose key (for a long time, we didn&#8217;t know it, but Norwegian and French users could not type } or { in NetBeans - kind of limits the usefuless of a Java IDE).</p>
</div>
<div class="paragraph">
<p>All standard shortcuts should be bound with wildcard keys - e.g., in the layer, not AS-P for Alt-Shift-P, but OS-P, which will map to Alt-Shift-P on PC and Ctrl-Shift-P on mac.  For Ctrl/Command, in the layer put DS-P instead of CS-P to bind conditionally to ctrl on PC and command on mac.</p>
</div>
<div class="paragraph">
<p>All Alt-bound keybindings in the IDE are specified with O so that they are rebound to Ctrl on mac, because they will interfere with typing in a lot of locales (the real way to think about it is "the mac does not have an Alt key" - it does but it is a composition key - you can&#8217;t use it).</p>
</div>
<div class="paragraph">
<p>Any alt-bound keybinding on mac is a bug.  If you use the logical syntax for keybindings, your app will always work intuitively on any platform.</p>
</div>
<div class="paragraph">
<p>Note that the mapping handling does skip key combinations that simply can&#8217;t work on Mac - for example, Command-H and Command-Q are always consumed by the OS, so D-H and D-Q map to Ctrl-H and Ctrl-Q, respectively, everywhere.</p>
</div>
</div>
